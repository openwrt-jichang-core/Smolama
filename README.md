# Ollama 集群扫描台

把原本的命令行扫描脚本改造成了一个可视化网页应用，主机地址可以在网页里动态添加/删除，
支持随时启动/停止扫描，扫描日志和结果实时显示。

## 功能

- **主机管理**：网页表单添加任意数量的 Ollama 地址（`http://ip:11434`），可随时删除，保存在持久化卷中，重启容器不丢失。
- **三阶段扫描**（逻辑与原脚本一致）：
  1. 发现各主机上的模型
  2. 快速可用性测试
  3. 高级代码生成测试（路径穿越防护 / 多方法类测试）
- **开始 / 停止按钮**：扫描可随时中断，不需要等它跑完。
- **实时日志**：网页轮询后端，滚动展示扫描过程。
- **结果面板**：按主机+模型分组展示每项测试的通过/失败情况。

## 在 Coolify 中部署

### 方式一：Docker Compose（推荐）

1. 在 Coolify 新建一个 **Resource → Docker Compose** 项目。
2. 把这个文件夹上传到你的 Git 仓库（或者用 Coolify 的「上传代码」功能），确保 `docker-compose.yml` 在根目录。
3. Coolify 检测到 `docker-compose.yml` 后会自动构建镜像。
4. 部署完成后，通过 Coolify 分配的域名/端口访问，默认容器内端口是 `8000`。
5. 数据卷 `ollama-scanner-data` 会持久化保存你添加的主机列表和最近一次扫描结果，重新部署不会丢失。

### 方式二：Dockerfile 直接构建

1. 在 Coolify 新建 **Resource → Application**，选择这个仓库，构建方式选 **Dockerfile**。
2. 设置端口映射：容器内 `8000` → 你想要的外部端口。
3. 挂载一个持久化卷到容器内的 `/data` 目录（Coolify 的 Storage / Volumes 设置里添加），否则主机列表在容器重建后会丢失。

### 环境变量

| 变量 | 说明 | 默认值 |
|---|---|---|
| `DATA_DIR` | 主机列表和扫描结果的存储目录 | `/data` |

## 本地测试（不用 Docker）

```bash
cd backend
pip install -r requirements.txt
uvicorn main:app --host 0.0.0.0 --port 8000
```

然后打开 `http://localhost:8000`。

## 使用说明

1. 打开网页后，在左侧「主机地址」面板输入 Ollama 地址（例如 `http://192.168.1.10:11434`），点击「+ 添加」。
2. 可以添加任意多个地址，每个都会显示在列表里，点击 × 可以移除。
3. 点击「▶ 开始扫描」启动三阶段扫描，日志会实时滚动显示。
4. 扫描过程中可以随时点击「■ 停止」中断。
5. 扫描全部完成（或被停止）后，下方「测试结果」面板会显示每个模型的通过情况。

## 关于安全

请只添加你自己拥有或已获得明确授权测试的 Ollama 服务器地址。这个工具会向目标主机发送真实的
推理请求（包括让模型生成代码并在扫描服务器本地执行测试），未经授权对他人服务器发起持续请求可能
违反相关服务条款或法律法规。

### 登录密码 / 防爆破

- 首次启动前请通过环境变量 `ADMIN_PASSWORD` 设置一个强密码；若不设置，容器会自动生成一个随机密码
  并**只打印在启动日志里**，请务必妥善保存，否则只能改环境变量或清空 `/data/auth.json` 重置。
- 未登录时访问网站会被重定向到 `/login.html`，所有 `/api/*` 数据接口在未登录状态下一律返回 401。
- 登录失败会按来源 IP 计数：前 3 次错误不锁定，从第 4 次起按指数退避锁定
  （2 分钟 → 4 分钟 → 8 分钟…上限 60 分钟），避免密码被暴力枚举。
- 会话通过服务端随机 token + `HttpOnly` / `SameSite=Strict` Cookie 下发，默认有效期 12 小时。
- 部署在有 HTTPS 的反向代理（Coolify 默认即是）之后请保持 `COOKIE_SECURE=true`（默认值）；
  只有纯本地 `http://` 调试时才需要设为 `false`。

### 防御滥用 / 防止主机被拿去跑挖矿等程序

- 全站接口有基于来源 IP 的进程内限流，超过阈值会返回 `429`，缓解刷接口/打流量式的滥用。
- 统一加了 `X-Frame-Options`、`Content-Security-Policy` 等安全响应头。
- 模型生成的代码在执行"高级测试"时运行在独立临时目录、`python3 -I`（隔离模式）子进程中，
  并对 CPU 时间、内存、进程数、文件大小设置了资源上限，降低失控/被投毒模型返回的代码
  (例如试图常驻挖矿或 fork 炸弹) 对宿主机造成影响的可能性。这是尽力而为的纵深防御，
  **不是**完整沙箱隔离，生产环境建议进一步用独立容器或 gVisor 级隔离运行代码执行部分。
- 容器默认以非 root 用户运行，并在 `docker-compose.yml` 中加了 `no-new-privileges` 与
  `cap_drop: ALL`，降低容器被攻破后进一步提权/横向移动的空间。

## 高级测试用例

高级测试阶段会让模型分别处理 5 类不同核心方向的题目，每类都刻意选取该方向里容易踩坑的边界情况：

1. **路径穿越防护**：文件读取函数是否正确拒绝 `../` 相对路径和绝对路径越权访问。
2. **多方法状态类（SiteManager）**：增删查改的异常处理与排序是否都正确。
3. **线程安全 LRU 缓存**：多线程并发读写下是否会出现竞态条件、死锁，以及淘汰顺序是否正确。
4. **带引号转义的 CSV 单行解析**：嵌套逗号、转义双引号、空字段等边界情况。
5. **精度安全的平均值计算**：忽略缺失值、四舍五入到银行家舍入、类型校验与空输入异常处理。

## 排行榜与连通性侧边栏

- 点击右上角「🏆 排行榜」展开排行榜面板。
- 左侧栏列出已添加的主机地址，点击某个地址会展开该主机下已发现的模型列表，
  每个模型前面的圆点显示连通性：绿色 = 正常，红色 = 失败，灰色 = 未知。点击某个模型会
  立即发送一条「你好」进行连通性重测，圆点会实时更新。
- 右侧是按高级测试总耗时升序排名的排行榜，只有全部测试通过的 主机+模型 才会参与排名；
  存在失败项的会额外记录在下方「失败记录」里，不参与排名。
- 每一行都有「↻ 重新测试」按钮，可以随时验证该 链接+模型 是否仍然可达、测试是否仍然通过。
